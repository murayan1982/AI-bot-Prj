# v1.5.0 Baseline for v1.6

## Purpose

This document records the stable v1.5.0 baseline before starting v1.6 work.

v1.6 focuses on configuration responsibility cleanup and developer experience improvements.
Before changing structure, the stable main flow, known issues, and refactoring boundaries are fixed here as a safety baseline.

---

## Stable Main Flow

The following flow is considered stable in v1.5.0 and should not be broken during early v1.6 work.

```text
User input
-> LLM raw response
-> parse_emotion_response()
-> emotion + clean_text
-> display clean_text
-> TTS clean_text
-> resolve_emotion_hotkey()
-> VTS trigger_hotkey()
```

This is the primary behavioral baseline for v1.6.

---

## Confirmed v1.5.0 Capabilities

At the v1.5.0 baseline, the framework already provides:

- preset-based runtime startup modes
- RuntimeConfig-based runtime initialization
- multi-LLM routing and fallback flow
- STT / TTS optional integration
- VTube Studio optional integration
- emotion tag parsing and clean text extraction
- character-level VTS hotkey mapping
- plugin-based emotion / VTS handling
- modular structure for runtime, pipeline, plugin, and config flow

This means the project has already reached the stage of a working developer-oriented framework core.

---

## Current Positioning at Baseline

At the v1.5.0 baseline, the project should be treated as:

- a developer-oriented framework
- not a finished consumer application
- already functionally established as an AI character development base
- still needing configuration cleanup, structure clarification, and developer-experience improvements

In other words:

- v1.5 established the core framework flow
- v1.6 will improve configuration responsibility clarity and maintainability
- v2.0 can later focus more strongly on usability, extensibility, and developer experience

---

## Known Issues at Baseline

The following issues are known at the v1.5.0 baseline:

- Long TTS playback may occasionally produce a wait timeout warning even when playback still works in practice.
- `tts_provider` naming and implementation behavior are not fully aligned.
- `settings.py` currently mixes multiple responsibilities such as defaults, registry-like values, legacy flags, and derived values.
- `models.py` still mixes registry-like definitions and environment-linked voice configuration concerns.
- Some README structure descriptions are slightly ahead of the actual file layout.
- Comments and docstrings are still limited in several responsibility-heavy files.

These are baseline cleanup targets, not immediate release blockers for the v1.5.0 state.

---

## Early v1.6 Refactoring Entry Points

The first cleanup targets for v1.6 are:

- `config/settings.py`
- `config/models.py`
- `config/loader.py` boundary clarification

These are the main entry points for configuration responsibility cleanup.

---

## Responsibility Cleanup Direction

v1.6 should move gradually toward the following responsibility model:

- RuntimeConfig as the runtime source of truth
- presets for startup mode definition
- characters for character-specific differences
- registries separated from general settings
- secrets separated from developer defaults
- legacy compatibility separated from the main runtime path

This cleanup should be incremental and should not become a large rewrite.

---

## Areas Explicitly Not Targeted in Early v1.6

The following are not primary early-v1.6 goals:

- redesigning the main emotion flow
- replacing the plugin system entirely
- major GUI work
- advanced local LLM support
- advanced streaming renderer redesign
- advanced motion control redesign
- character editor UI

These may be future topics, but they are outside the scope of early v1.6 cleanup.

---

## Safety Rules for v1.6

- Do not break the stable main flow.
- Treat RuntimeConfig as the runtime source of truth.
- Keep `text_chat` as the safe default for development.
- Separate responsibilities gradually instead of performing a large rewrite.
- Prefer small, reversible refactors.
- Do not mix cleanup work with large feature expansion.

---

## Comment / Docstring Policy

Comment and docstring improvement is part of the long-term project policy.

For v1.6:

- Add comments/docstrings incrementally in files touched during each day's work.
- Prioritize responsibility, intent, and boundary explanation over obvious line-by-line code explanation.
- Use a later consistency pass to review missing docstrings, responsibility notes, and comment quality.

This policy supports framework maintainability and readability without creating excessive comment noise.
